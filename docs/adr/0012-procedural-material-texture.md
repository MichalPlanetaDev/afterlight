# ADR 0012: Persistent procedural material texture

## Context

P11 established correct attachment ownership and P10 established device-local transfer infrastructure, but the aperture still derived all surface response from vertex colour. The renderer had no sampled-image path, no persistent material descriptor set and no texture upload contract.

The first texture also needs to remain project-owned, deterministic and reproducible without importing an asset that bypasses the planned content pipeline.

## Decision

P12 introduces one persistent `MaterialTexture` resource owned by the swapchain renderer but independent of swapchain extent. It owns the optimal-tiled device-local image, memory, image view, sampler, descriptor-set layout, descriptor pool and descriptor set.

Scene data remains in frame-local descriptor set zero. Persistent material state occupies descriptor set one. Binding zero is a sampled image and binding one is a sampler. This separation reflects update frequency and lifetime rather than placing unrelated resources in one owner.

The aperture calibration texture is generated with integer arithmetic from versioned dimensions and ring parameters. Concentric divisions, radial spokes, an optical hub and a bounded roughness channel produce a stable material field. FNV-1a identifies the exact generated byte payload.

Upload uses host-visible staging memory with coherent preference and explicit non-coherent flushing. The destination is an `R8G8B8A8_SRGB`, optimal-tiled, device-local image. Synchronization2 transitions it from undefined to transfer destination and then to shader-read-only layout. A fence completes the transfer before staging resources are released.

The sampler uses linear minification and magnification, one explicit mip level, clamp-to-edge addressing and no anisotropy because sampler anisotropy is not enabled in the current logical-device feature set.

## Consequences

The graphics pipeline now consumes two descriptor-set layouts and binds both sets for each draw. The shader samples material colour and roughness while preserving the established directional-light, exposure and geometry contracts.

The resource is intentionally immutable after construction and survives swapchain recreation. Mip generation, compressed runtime formats, imported texture metadata, material schemas and offline asset compilation remain later milestones.
